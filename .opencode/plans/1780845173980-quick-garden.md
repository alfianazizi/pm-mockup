# Project Management Dashboard Prototype Implementation Plan

> **For OpenCode:** REQUIRED SUB-SKILL: Use `executing-plans` to implement this plan task-by-task. Use `test-driven-development` where practical for domain helpers/actions, and `verification-before-completion` before claiming the implementation is complete.

**Goal:** Build a frontend-only enterprise project management and budget monitoring dashboard prototype for a holding company, with the deepest interactions around Project Templates, Project creation, Project detail, and Project/Milestone approvals.

**Architecture:** Preserve the existing Better-T-Stack pnpm monorepo, React/Vite web app, TanStack Router file routing, Tailwind v4 CSS-first styling, and shared `packages/ui` primitives. Add app-domain types, seeded mock data, localStorage-backed state, role/scoped selectors, and mutating action helpers under `apps/web/src`. Use the provided corporate light palette with a dark sidebar and Recharts for primary dashboard/budget charts.

**Tech Stack:** React, Vite, TypeScript, TanStack Router, Tailwind v4, pnpm workspaces, `@project-management-mockup/ui`, Recharts, localStorage.

---

## Resolved Product Decisions

- Keep TanStack Router and the existing monorepo instead of switching to React Router.
- Treat a Subholding as a business unit under the Holding, with projects, users, budgets, and rollups.
- Give each Subholding an approved budget allocation, but keep detailed period budgeting secondary.
- Prioritize Project Templates, Project Wizard, Project List, Project Detail, and Approvals; keep Dashboard, Budget Monitoring, Reports, Users, and Settings polished but lighter/read-mostly.
- Create Projects from Project Templates by snapshot-copying milestones and steps; keep `templateId` and template version for traceability.
- Steps always belong to exactly one Milestone.
- Derive Project progress from Milestone/Step progress.
- Track Milestone work status separately from Approval status.
- Make only Project approval and Milestone approval deeply interactive.
- Use a simple Project lifecycle: `Draft` -> `Waiting Approval` -> `Active` -> `Completed`; use `On Hold`, `Delayed`, and `Cancelled` as realistic examples without full workflows.
- Hide unauthorized menu items and guard direct route access.
- Filter visible data by demo user scope.
- Let Holding Admin manage global Project Templates; Subholding Admins and Project Owners can use templates but not edit the global library.
- Let Holding Admin create Projects for any Subholding; Subholding Admin creates Projects for their own Subholding; Project Owner manages assigned Projects.
- Assign approval requests to specific demo Approver users.
- Persist mock interactions in localStorage over seeded mock data.
- Use the `PROJECT.md` light corporate palette with a dark sidebar.
- Optimize for desktop/tablet stakeholder presentation with usable mobile fallback.
- Display money as IDR in billions, e.g. `Rp 12.5B`.
- Use simple Project Template version labels, e.g. `v1.0`.
- Use soft archive behavior: hide archived records by default, preserve history.
- Build Project Template create/edit as a single sectioned editor.
- Build Project creation as a five-step wizard.
- Make `/` redirect to `/login` or `/dashboard` based on mock auth state.
- Use named demo user cards on `/login`.
- Compute Risk Level from simple signals with optional seeded override.
- Define Budget Status as `Normal` below 80%, `Near Limit` from 80% through 100%, and `Over Budget` above 100% actual spending against approved project budget.
- Use calendar-year periods: quarters start in January; semesters are Jan-Jun and Jul-Dec.
- Implement attachments as fake upload metadata only.
- Seed Activity Logs and append entries for key local interactions.
- Use real Recharts charts for main dashboard/budget views.
- Make Subholding and Year/Period global header controls for reporting/scoped pages.
- Approval decisions update related Project/Milestone state.
- For Milestone decisions, `Revision Required` moves work status back to `In Progress`; `Rejected` keeps work status `Submitted` with approval status `Rejected`.

## Key Risks

- TanStack Router file names must match the starter's file-route conventions; let tooling regenerate `routeTree.gen.ts`.
- Template snapshot creation must deep-copy nested milestones/steps to avoid accidental mutation from later template edits.
- localStorage can mask seed-data changes during development; a reset-to-seed action is required.
- Role-based access must include both menu hiding and direct-route guards.
- Approval logic is the highest-risk behavior because Project lifecycle, Milestone work status, and Approval status are intentionally separate.
- The provided palette should be mapped into existing Tailwind v4/shadcn tokens instead of hardcoding colors throughout components.
